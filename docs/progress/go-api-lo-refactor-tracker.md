Go `/api/v1` 接口的 `lo` 专项改造已经完成。归档批次 B01-B14 均已收口，标记为“不建议改造”的接口是兼容保留 no-op：

本文档不再维护逐接口改造状态。后续新增接口时，不需要继续追加批次表，而应按本文件的使用准则判断是否引入 `lo`。

## 2. 保留准则

`lo` 可以用，但只能用在“更清楚”，不能用在“更花”。

推荐使用：

- `Map` / `Filter` / `Uniq` / `SliceToMap` 等清晰集合转换。
- 替代无业务含义的样板循环。
- 处理去空、去重、映射、集合判断等简单数据整理。

不推荐使用：

- 隐藏副作用。
- 把复杂业务判断塞进匿名函数。
- 为了“全项目统一 lo”替换本来更直白的 Go 原生代码。
- 在错误处理、事务、副作用控制中追求链式写法。

判断标准只有一个：同事第一次读到这里，是否更快看懂真实业务意图。

## 3. 长期约束

后续涉及集合处理的改动，必须同时满足：

- 不改变 HTTP、CLI、脚本调用方的黑盒契约。
- 不改变错误语义和 `dry-run` 语义。
- 不引入跨层依赖。
- 不因为使用 `lo` 增加数据库查询次数。
- 不为了短代码牺牲可调试性。
