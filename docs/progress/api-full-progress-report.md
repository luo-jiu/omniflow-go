# Go API 与日志改造归档摘要

更新时间：2026-04-15  
状态：已完成，归档保留

## 1. 当前结论

`omniflow-go` 的 `/api/v1` 接口实现、规范收口和日志 P1 接入已经完成。

| 指标 | 结果 | 说明 |
|---|---:|---|
| `/api/v1` 接口总数 | 45 | 以当前路由注册为准 |
| 功能实现 | 45/45 | 全量可用，含兼容 no-op 1 个 |
| 规范收口 | 45/45 | 分层边界、依赖显式错误语义、最小必要 `lo` 已完成 |
| 日志 P1 | 45/45 | 请求日志、业务日志、必要审计日志已接入 |

该文档不再维护逐接口“已完成”流水账。新增或调整 API 时，应更新更贴近当前事实的契约文档、路由文档或对应进度台账。

## 2. 保留规则

后续新增或修改 API 时，仍然必须遵守：

- 保持 Go 当前对外契约稳定，除非需求明确要求变更。
- 响应外壳保持 `code/message/data/request_id`。
- handler 只做参数绑定、响应封装、错误码映射。
- usecase 负责业务编排、权限、事务、审计和 `dry-run`。
- repository 收敛 SQL、缓存、对象存储细节。
- 写链路必须区分 `mode=dry-run` 与 `mode=execute`。
- 请求日志保留 `actor/error/dry-run/request_id` 等关键字段。
- 业务日志使用结构化字段，不拼接不可检索的大字符串。

## 3. 归档批次摘要

| 批次 | 范围 | 结果 |
|---|---|---|
| P0 | 日志底座 | JSON/文本、console/file、滚动切割、严格配置校验已完成 |
| P1 | 全接口日志接入 | Auth/User/Library/Node/Directory/File/Tag/Health 已覆盖 |
| 规范收口 | API 分层与错误语义 | 依赖缺失显式错误、兼容 no-op、最小必要 `lo` 已完成 |

详细逐接口记录可从 Git 记录查看，当前文档只保留长期有效结论。

## 4. 后续建议

后续进入日志 P2 时，优先做这些长期收益项：

1. 输出日志字段字典，统一 `resource_id`、`mode`、`result_count` 等命名。
2. 增补关键链路日志测试，至少覆盖 auth、node mutation、directory upload。
3. 规划 stdout/file 到 collector 的采集接入，保持业务代码零侵入。

## 5. 维护规则

只有出现以下情况才需要更新本文档：

- `/api/v1` 总接口数量发生变化。
- 日志接入策略发生全局变化。
- API 规范收口状态从“已完成”回退或重新开启专项改造。
