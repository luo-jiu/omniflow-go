# OmniFlow Java -> Go 迁移交接说明

更新时间：2026-04-08  
适用对象：熟悉 Java 旧服务、即将参与 Go 重构和联调的开发/Agent

## 1. 这份文档解决什么问题

本项目不是“按 Java 代码逐行翻译”，而是“按 Java 接口黑盒契约重写”。  
目标是前端无感切换：接口路径、参数和返回语义保持一致；底层实现允许彻底重构。

这份文档用于统一三件事：
- Go 版本与 Java 的关键差异是什么。
- 目前已经做到了什么程度。
- 后续继续开发必须遵守哪些规范，避免返工。

## 2. 当前迁移状态（结论先行）

- Java 接口总数：41
- Go 已实现并完成黑盒对齐：41
- 未实现：0
- 当前阶段定位：后端已进入“可前端联调”状态

说明：

- 前端联调阶段仍可能出现少量边界差异，需要按黑盒行为继续微调。

## 3. 与 Java 的核心关系：什么必须一样，什么可以不一样

### 3.1 必须一样（硬约束）

- Method + Path 保持一致。
- 请求参数名、大小写、默认行为保持一致。
- 响应外壳保持一致（`code/message/data/request_id`）。
- 关键业务语义保持一致（鉴权、权限、分页、排序、错误语义）。

### 3.2 可以不一样（允许重写）

- 数据模型与表结构可以变化。
- SQL 写法可以变化（ORM / gen / Raw SQL 均可，按规范选）。
- 事务实现与并发控制方式可以变化。
- 存储实现可以变化（默认 MinIO，但保留 provider 切换入口）。

## 4. 当前 Go 版本相对 Java 的关键实现差异

### 4.1 数据库与树模型

- 已从 MySQL 路线切到 PostgreSQL 路线。
- 节点层级采用邻接表思路，不走 Java 旧实现的原路径。
- 对树查询和路径类能力，使用 PG 友好的递归/分层查询策略实现。

### 4.2 节点行为对齐重点

- `create node`：父节点缺失或失效时，回退到库根节点。
- `move node`：使用间隔排序策略，必要时重排，处理并发窗口。
- `PATCH /nodes`：按 Java 兼容语义保留空实现。
- 读链路（descendants/children/ancestors/path/root）补齐权限校验。
- root 节点支持坏父引用自修复，避免历史脏数据导致链路异常。

### 4.3 用户与头像

- 用户名检查语义与 Java 对齐：`/user/exists` 返回“是否可用（未被占用）”。
- 头像 URL 使用预签名策略，默认短期有效。
- 头像上传采用“适中校验”：
  - 仅用于头像场景，不影响通用文件系统逻辑。
  - 支持文件头识别与 `image/*` 规则（排除不安全类型），并保留常见扩展名兜底。

### 4.4 标签与状态字段

- tag `enabled` 已按 PG 布尔映射语义修正并对齐接口表现。
- `search-types` 返回语义按 Java 黑盒行为对齐（兼容前端依赖）。

### 4.5 对象存储

- 默认 provider 仍是 MinIO。
- provider 切换入口已预留（如 s3/oss/cos/obs），未实现 provider 会明确报错而不是静默回退。

## 5. 分层与目录规范（执行基线）

当前采用轻量分层，依赖方向是：

`transport -> usecase -> domain(port) <- repository(impl)`

职责边界：
- `transport`：只做协议绑定、响应映射，不写业务规则。
- `usecase`：业务编排、权限、审计、事务边界；不直接操作 DB/Redis/MinIO 客户端。
- `domain`：稳定业务对象和端口抽象；不依赖具体基础设施。
- `repository`：具体存储实现（Postgres/Redis/Object），封装查询细节和错误映射。

目录组织原则：
- 先按仓储类型拆：`postgres` / `redis` / `object`
- 再按业务域拆：`user` / `library` / `node` / `tag`
- 路由按资源拆文件，不把所有资源堆进单文件

## 6. SQL 与事务规范（后续必须继续遵守）

### 6.1 SQL 规范

- 常规 CRUD 优先使用 gorm/gen query。
- 复杂场景（递归、复杂聚合）允许 Raw SQL。
- Raw SQL 集中在专用 SQL 文件，不散落在业务方法体。
- 禁止字符串拼接构造 SQL 条件，统一参数化。

### 6.2 事务规范

- 事务边界在 usecase 层定义。
- usecase 通过 `Transactor.WithinTx` 开启事务。
- repository 通过 `ctx` 感知并复用事务；无事务时自动回落默认连接。
- 避免业务层显式串联 `WithTx`，优先上下文感知方案。

## 7. 鉴权与会话规范

- HTTP 鉴权采用中间件统一校验。
- 白名单使用“精确路径匹配”，仅放公开接口（健康、登录、注册/用户名检查、公开上传链路）。
- Session 能力已下沉到 Redis 仓储实现，usecase 不直连 Redis 客户端。

## 8. 代码风格与提交规范（避免风格漂移）

- handler/usecase 对外方法保留简洁中文注释，先讲业务意图再讲参数。
- 新增逻辑优先按资源拆文件，不写“超大综合文件”。
- 出错语义优先对齐 Java 黑盒结果，再考虑内部错误文本美化。
- 每次提交保持“小步可回滚”，优先一个行为点一个提交。
- 只在必要处使用 Raw SQL，其余保持统一风格，避免“同类逻辑多种写法”。

## 9. 给后续 Agent 的工作方式建议

推荐流程：
1. 先看接口契约与本交接文档，明确“黑盒行为”目标。
2. 判断是否有基建阻塞（端口/仓储/事务/配置）；有阻塞先补基建。
3. 无阻塞时再改业务逻辑，改完立即做最小验证。
4. 对齐结果回写状态文档，保持迁移进度透明。

判断优先级：
- 先保证“前端可感知行为”正确。
- 再优化内部实现质量（SQL 风格、拆包、注释、复用）。
